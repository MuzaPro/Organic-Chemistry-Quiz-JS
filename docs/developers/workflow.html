Branching Strategy

Use the following branching strategy for all contributions to the project:

  • main - Production branch, contains stable code
  • development - Integration branch for testing features
  • feature/feature-name - Feature branches for new development
  • bugfix/bug-name - Bug fix branches

Git branching strategy visualization

Never commit directly to main or development. Always create a feature or bugfix branch and submit a pull request.

Workflow Steps

  1. Create a new branch

    git checkout development
git pull
git checkout -b feature/your-feature-name

    When naming your branch, use descriptive names that indicate what you're working on, for example:

    • feature/audio-volume-control
    • bugfix/mobile-drag-issue

  2. Make changes to the codebase

    Implement your feature or bug fix according to the ticket requirements. Follow the Code Style Guidelines for consistency.

  3. Commit your changes

    git add .
git commit -m "Meaningful commit message"

    Write clear, descriptive commit messages that explain what changes were made and why. For example:

    • "Add volume control slider to audio system"
    • "Fix drag event handling on mobile devices"

  4. Push your branch

    git push -u origin feature/your-feature-name

  5. Create a pull request

    Go to the GitHub repository and create a pull request to merge your changes into the development branch. Include:

    • A clear title describing the change
    • A description of what was changed and why
    • Reference to any related issues or tickets (e.g., "Fixes #27")
    • Screenshots or GIFs if the change affects the UI

  6. Address review comments

    Respond to any feedback provided by reviewers. Make requested changes on your branch and push additional commits.

  7. Merge into development

    Once approved, your branch will be merged into development. This step is typically done by a project maintainer.

  8. Test in development

    Verify that your changes work as expected in the integrated development branch before they're considered for merging to main.

Creating New Features

When adding new features to the quiz game, follow these guidelines:

1. Plan Your Feature

  • Create a ticket describing the feature
  • Outline the requirements and acceptance criteria
  • Identify which existing components will be affected
  • Plan new components if needed

2. Code Organization

  • Create new files for significant new features
  • Follow the existing module pattern
  • Keep related functionality together
  • Document public APIs

3. Integration Steps

  1. Create new modules if needed: 
    const NewFeature = (() => {
  // Private variables and functions
  
  // Public API
  return {
    initialize: () => { /* Implementation */ },
    publicMethod: () => { /* Implementation */ }
  };
})();
  2. Register with main application in app.js: 
    document.addEventListener('DOMContentLoaded', () => {
  // Initialize existing components
  
  // Initialize new feature
  NewFeature.initialize();
});
  3. Update HTML to include new UI elements if needed
  4. Add CSS for new components

4. Documentation

Always document your new feature:

  1. Update relevant documentation files
  2. Add JSDoc comments to functions
  3. Create a feature-specific documentation file if substantial

Ticket System

The project uses a ticket system for tracking work. Each ticket should include:

Ticket Format

# Ticket #[Number]: [Title]

**STATUS:** OPEN/IN PROGRESS/CLOSED
**Priority:** High/Medium/Low
**Type:** Feature/Bug/Enhancement
**Estimate:** [Time Estimate]
**Assigned to:** [Developer Name]

## Description
[Detailed description of the task]

## Tasks
1. [Specific task 1]
2. [Specific task 2]
...

## Acceptance Criteria
- [ ] [Criterion 1]
- [ ] [Criterion 2]
...

## Related Files
- [File path 1]
- [File path 2]
...

## Implementation Notes
[Notes for implementation, if any]

Ticket Workflow

  1. Create a new ticket for each feature or bug
  2. Assign the ticket to a developer
  3. Update the status to IN PROGRESS when work begins
  4. Create a feature branch based on the ticket number
  5. Reference the ticket number in commit messages
  6. Update the ticket with progress notes
  7. Mark acceptance criteria as completed
  8. Change status to CLOSED when completed

Pull Request Guidelines

When creating pull requests, follow these guidelines to ensure smooth reviews and integrations:

Before Submitting a Pull Request

  • Ensure all tests pass
  • Verify that your code follows the style guidelines
  • Check for any console errors or warnings
  • Test your changes across different browsers and screen sizes
  • Make sure all acceptance criteria are met

What to Include in a Pull Request

  • A clear and descriptive title
  • A detailed description of the changes
  • References to relevant issues or tickets
  • Screenshots or GIFs for visual changes
  • Notes about any breaking changes
  • Any special testing instructions

Communication Tip

When working on features, regular communication helps avoid duplicate work and ensures alignment with project goals. Don't hesitate to ask questions in pull request comments or via project communication channels.
